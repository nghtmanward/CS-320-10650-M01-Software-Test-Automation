Project Two Artifact — Summary & Reflections Report
This written report summarizes my testing approach, alignment with requirements, test coverage, and reflections on the mindset and discipline required in software testing.

Reflection
How can I ensure that my code, program, or software is functional and secure?
I ensure functionality and security by validating all inputs, enforcing strict data rules, and writing unit tests that confirm each requirement is met. Defensive programming helps prevent invalid data from entering the system, while exception handling ensures the software reacts safely to unexpected conditions. Automated tests allow me to verify behavior consistently and catch regressions early.

How do I interpret user needs and incorporate them into a program?
I translate user needs into clear, testable requirements. In this course, the specifications acted as user stories, and I used them to guide both the implementation and the tests. By turning each requirement into a corresponding unit test, I ensured the final code aligned with what the user actually needed and prevented feature drift.

How do I approach designing software?
I start by identifying the core classes, responsibilities, and constraints. From there, I design small, focused methods that are easy to test and maintain. I rely on test‑driven thinking to shape the design: if I can write a clear test for a behavior, I can design the code to satisfy it. This leads to modular, predictable, and maintainable software.
